# 作品集整站重设计规格

## 目标

重新设计当前 React + Vite 简历网站，让它从普通简历页升级为“未来感个人品牌 + 高级技术简历”的作品集页面。

整站需要同时满足：

- 第一眼有个人品牌感，能记住“金敏睿 / AI 前端开发工程师”。
- 内容仍然服务招聘阅读，能快速看出经验、项目、能力和联系方式。
- 视觉和动效比当前更丰富，但不做工作台、Dashboard 或信息卡堆叠式页面。
- 页面叙事从“身份”推进到“AI 研发流程方向”，再用证据、项目和经历托底。

本规格覆盖之前的首屏重设计规格：`docs/superpowers/specs/2026-07-01-portfolio-hero-redesign-design.md`。后续实现以本整站规格为准。

## 当前上下文

项目是单页作品集，核心文件是：

- `src/App.jsx`
- `src/styles.css`

当前页面已经包含：

- Hero 身份介绍。
- quick facts 指标区。
- AISDLC workflow tab。
- evidence 能力证据。
- cases 项目案例。
- skills 技能标签。
- timeline 经历。
- contact 联系方式。

当前版本已经使用 GSAP 和 ScrollTrigger，并已有首屏入场、滚动 reveal、Hero 背景流动、鼠标 tilt 和按钮高光。

用户已确认的新方向：

- 除首屏外，其他模块也需要重新设计。
- 整站风格采用“未来感个人品牌 + 高级技术简历”。
- 内容结构适度重排，而不是完全保留或大幅重组。
- 首屏不使用左右布局，不展示电话、邮箱等详细信息。
- 不做 AI 工作台、SaaS 控制台、后台系统或 Dashboard 风格。

## 非目标

- 不新增路由、多语言、下载简历或后台能力。
- 不改写履历事实，不新增未经确认的项目或数据。
- 不引入新的动画库。
- 不做深色科幻站。
- 不做锁屏滚动、长时间 pinning 或必须按固定节奏观看的叙事页面。
- 不为了视觉效果牺牲移动端可读性。

## 页面顺序

采用适度重排后的整页结构：

1. **Hero：未来感个人品牌封面**
2. **Proof Strip：核心事实刻度**
3. **AISDLC Narrative：AI 参与研发流程**
4. **Evidence：能力证据墙**
5. **Cases：项目案例陈列**
6. **Skills + Timeline：能力栈与经历背景区**
7. **Contact：收尾联系区**

这个顺序让页面先建立差异化，再证明可信度，最后给出完整履历和行动出口。

## 视觉系统

整体基底保持浅色、干净、专业。视觉语言使用低透明度网格、细线轨迹、蓝色和青色高光、坐标点、扫描光和轻量玻璃质感。

设计边界：

- 页面可以有未来感，但不能像后台控制台。
- 可以使用抽象动态视觉，但不能让装饰抢过正文。
- 卡片可以存在，但不能让整站变成卡片堆叠。
- 背景可以有科技感纹理，但主体内容必须清晰可读。
- 蓝色是主强调色，青色可作为辅助光效，避免整页只有单一蓝色层次。

排版上，Hero 使用更大的标题尺度；内容区使用更紧凑、适合扫描的标题和正文尺度。页面应有明显节奏变化：首屏开阔，流程区有叙事，案例区信息密度更高，经历区更克制。

## 模块设计

### Hero：未来感个人品牌封面

Hero 不再使用左右布局。姓名和身份位于中心或视觉中心附近，形成首屏主焦点。

内容包含：

- `金敏睿 / Minrui Jin`
- `AI 前端开发工程师`
- 一句压缩后的摘要，强调把 AI 接入需求理解、工程实现、Review 和知识沉淀。
- 保留主要行动按钮，不增加新功能。

背景为全屏动态视觉场，包含微弱网格、细线轨迹、坐标点、扫描光和低透明度关键词。关键词如 `AISDLC`、`Agent`、`MCP`、`Frontend`、`Review`，只作为视觉层，不承担主阅读任务。

首屏不展示电话、邮箱等详细联系方式。

### Proof Strip：核心事实刻度

Proof Strip 可以作为 Hero 底部或 Hero 后的第一段轻量内容，展示 3-4 个事实：

- `6 年前端`
- `美团 / 小米`
- `AISDLC`
- `性能优化 50%`

它应像坐标刻度或封面注脚，而不是普通指标卡。作用是快速建立可信度，并引导继续向下滚动。

### AISDLC Narrative：AI 参与研发流程

保留现有 workflow 内容，但视觉上从 tab 控件升级为流程叙事区。

设计重点：

- 展示“需求澄清、方案评估、代码重构、Review / 排障”的阶段关系。
- 可以保留点击切换交互。
- 当前阶段需要有清晰高亮。
- 右侧或下方展示阶段标题、正文和产出。
- 视觉上像研发流程叙事，不像表单控件或后台步骤条。

动画上，该区是整页第二重点。进入时流程线和阶段节点依次出现，切换阶段时标题、正文、产出做短过渡。

### Evidence：能力证据墙

保留 4 条证据：

- 能接住复杂业务。
- 会盯关键链路体验。
- 能从 0 到 1 推功能落地。
- 在建设前端 AISDLC。

视觉上做成证据墙，强调每条证据的 `title / body / proof` 层级。proof 是最强的可信度锚点，需要比正文更醒目。

动效以滚动 stagger 和 hover 边线高光为主，不做持续动画。

### Cases：项目案例陈列

保留三个项目案例：

- 青云聚信。
- 校园运力 SaaS 小程序。
- 前端 AISDLC 建设。

每个案例以编辑式陈列呈现，突出：

- 项目名称和场景。
- 问题。
- 我的贡献。
- 结果。
- 技术栈。

案例区应比证据区信息密度更高，但需要通过排版分区提高可扫读性。hover 或滚动进入可以有轻量层次变化，但不能让内容难读。

### Skills + Timeline：能力栈与经历背景区

技能和经历合并成一个更紧凑的背景能力区，避免页面后半段分散。

设计方向：

- 技能标签可以做成技术云或分组标签，但不新增复杂分类。
- 时间线保留现有经历事实。
- 视觉上比 Hero、AISDLC、Cases 更克制。
- 重点是让招聘方快速确认技术栈和工作经历。

### Contact：收尾联系区

联系区作为最终行动出口，集中展示：

- 邮箱。
- 电话。
- GitHub。

视觉上与 Hero 呼应，可以使用类似的细线、坐标或高光，但动效保持短促。首屏移除的详细联系方式在这里完整出现。

## 动效分级

动效按重要度分级：

1. Hero 最丰富：背景视觉场、标题入场、关键词漂浮、扫描光、鼠标视差。
2. AISDLC 和 Cases 次之：流程节点进入、内容切换、案例层级 reveal。
3. Evidence、Skills、Timeline 只做节奏增强：滚动 stagger、轻微 hover。
4. Contact 做短促收尾动效。

所有连续动画都必须慢、轻、低透明度。动效不能改变内容阅读顺序，不能造成布局跳动。

`prefers-reduced-motion: reduce` 下：

- 关闭连续循环动画。
- 关闭鼠标视差。
- 关闭大幅位移和扫描光。
- 保留静态布局和必要的低强度入场。

## 技术设计

继续使用现有 `GSAP + ScrollTrigger`，不新增动画依赖。

React 层面：

- 允许重排现有模块顺序。
- 保留现有内容数据，不新增未经确认的数据。
- 重组 Hero DOM。
- 将 workflow 区从普通 tab 视觉升级为流程叙事结构。
- 将 skills 和 timeline 合并展示。
- 使用 `useRef` 管理需要动画的区域。
- 使用 `gsap.context()` 限定动画作用域。
- 在 cleanup 中清理 timeline、ScrollTrigger 和鼠标监听。

CSS 层面：

- 重建整站视觉系统，包括全局背景、模块间距、标题层级、浅色科技感纹理和高光样式。
- 使用响应式约束保证移动端可读性。
- 避免大面积卡片套卡片。
- 避免文字和按钮重叠。
- 避免横向溢出。
- 只对需要动画的元素添加 `will-change`。

## 验收标准

1. 整站不再像普通简历卡片堆叠。
2. 首屏不使用左右布局，不展示电话、邮箱等详细信息。
3. 页面整体能表达未来感个人品牌，同时保持招聘阅读效率。
4. 模块顺序符合 Hero、Proof Strip、AISDLC、Evidence、Cases、Skills + Timeline、Contact。
5. AISDLC 区更像流程叙事，而不是普通 tab 控件。
6. Evidence 区能快速看出能力证据和 proof。
7. Cases 区能清晰阅读问题、贡献、结果和技术栈。
8. Skills 和 Timeline 合并后仍能快速扫描技术栈和经历。
9. Contact 区完整展示邮箱、电话和 GitHub。
10. 桌面端动画层次比当前丰富，但不呈现工作台或 Dashboard 感。
11. 移动端无横向溢出、文字重叠或按钮溢出。
12. `prefers-reduced-motion: reduce` 下没有持续运动和明显视差。
13. `npm run build` 通过。
